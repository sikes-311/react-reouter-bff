# react-reouter-bff

## 比較表（詳細）

| 案                                        | 構成                                 | 学習コスト                                                               | SEO / SSR                        | BFFスケール                              | 開発チームスケールのしやすさ                                                                                                                                         |
| ----------------------------------------- | ------------------------------------ | ------------------------------------------------------------------------ | -------------------------------- | ---------------------------------------- | -------------------------------------------------------------------------------------------------------------------------------------------------------------------- |
| **Next.js 単体**                          | Next.js が UI + SSR + 軽量BFF を兼務 | △<br>概念が多く（Server Components, routing, cache）、初学者にはやや複雑 | ◎<br>SSR/SSG/ISRが標準で利用可能 | △<br>BFFとUIが同一プロセスで分離しづらい | △<br>ドメインごとにチームを分けても、共通の app 配下や server action / route handler に責務が集まりやすく、変更衝突やレビュー境界の曖昧さが出やすい                  |
| **React Router v6 + Spring Boot**         | フロントとBFFを完全分離              | ◎<br>Reactの基本に集中でき理解しやすい                                   | △<br>SSRは後付け設計が必要       | ◎<br>BFFだけ独立スケール可能             | ○<br>UIチームとBFFチームを分けやすく、ドメイン単位のAPI境界も引きやすい。ただしフロント側が単なるSPA構成だと、画面単位のデータ責務が曖昧になりやすい                 |
| **React Router v7 + Spring Boot（推奨）** | Data Modeで段階的に構築              | ◎<br>段階的に理解でき最も学習しやすい                                    | ○<br>必要に応じてSSR導入可能     | ◎<br>BFF独立スケール可能                 | ◎<br>ルート単位でデータ責務を置けるため、ドメイン単位で画面・API・BFF責務をそろえて分けやすい。複数チーム開発でも影響範囲を閉じ込めやすい                            |
| **Next.js + Spring Boot**                 | Next.jsはUI/SSR、BFFはSpring Boot    | ○<br>Next.jsの学習は必要だが分離は明確                                   | ◎<br>Next.jsのSSRを活用可能      | ◎<br>BFFは独立スケール可能               | ○<br>BFF側はドメイン単位で分割しやすいが、フロント側はNext.jsの機能が強力な分、各チームがServer ComponentやRoute Handlerに業務寄り処理を書き始めると境界が崩れやすい |
